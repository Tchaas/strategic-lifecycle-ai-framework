The model translates executive strategy into traceable, implementation-ready architecture. Forecasts are entered at progressively finer altitudes (objective → case → discovery); actuals are entered once at implementation and resolve both upward to the objective and sideways to value streams. The traceability spine — case → feature → capability → value stream / key activity — is what every linking table exists to preserve.

The flow

What leads to what
Sign up / sign in → Company Profile (workspace) → Departments → Strategic Objective → Business Architecture (optional) → Value Streams · Key Activities · Capabilities ⟶ Phase 2 ⟶ Lean Business Case → Discovery → Features · Requirements → Conceptual Deliverables → Implementation (actuals)
Forecast — 3 altitudes

Strategic Objective (hypothesis, integer metrics + financial_impact) → Lean Business Case (forecast_cost/value) → Discovery (qualitative refinement).

Actuals — entered once

At Implementation, per value stream (implementation_value_streams.allocated_cost/value). Objective actuals are computed, never stored.

Governance

Deliverables are suggestions: generated from data, edited by a human, saved as final (suggested → user_finalized). The AI layer plugs into this same gate with zero schema change.

Deliberate changes from the original board — verify these were intended: (1) Business Architecture is now company-level (one per workspace), not objective-owned — its create route moved to /workspaces/:id/business-architecture. (2) lean_business_cases is a new table (the board had its API but no DDL). (3) business_impact_links is retired — impacts now carry an optional linked_value_stream_id + linked_lean_business_case_id. (4) Single success_metric text fields on the objective are replaced by the strategic_objective_metrics child table (integers). (5) Auth is single-provider on users (the board's users had no password_hash). (6) users table name standardized (board referenced user_profiles in one FK).
AI Assistance Layer — additive, stateless, cost-aware. The AI that helps users fill in each component is a set of backend-mediated endpoints (/ai/draft, /ai/refine, /ai/suggest-links) over the unchanged 33-table schema — no new tables, because suggestions live only in the browser until the user saves through the normal entity endpoints. Cost is controlled by design: every call is user-triggered (nothing fires on typing), one structured call drafts a whole component rather than one call per field, a cheap model handles routine drafting with a larger model reserved for heavy synthesis, and prompt caching + lean (linked-records-only) context keep input tokens small. See the AI Assistance Layer API group and the AI assistance rules.

Origin & reuse — one architecture, two entry points

    
The same company-level architecture can be stood up from the strategic phase or from discovery. The origin flag (architecture | discovery) records which path created each component and is provenance only — it never scopes the component. Because every component is workspace-scoped (never tied to the discovery or case that created it), anything built in discovery is reusable by any future objective: value streams and capabilities through the objective's own selection junctions, the finer-grained components through their value streams and the objective's cases.

    

  


  

    
AI Assistance Flow

    
How the AI layer adds details — without adding tables

    
The AI layer doesn't appear in the entity-relationship diagram because it is stateless and adds no tables. It is a request/response flow between the browser, the FastAPI backend, and the AI provider. A suggestion populates the editable fields; only the user's Save writes anything to the database.

one structured call per component · cheap model for drafts, larger model only for deliverables

    

      
User-triggered only
No calls fire on typing or focus. Every call is an explicit click — the primary cost control, and the reason there is no background spend.

      
Stateless
The suggestion lives only in the browser. Nothing is persisted until the user saves through the normal entity endpoint — which is why AI adds zero tables.

      
AI never writes
AI only pre-fills editable fields; the user edits in place, and only Save persists the values. The same path the deliverable suggested → user_finalized gate already follows.

API Contracts

    
Endpoints, methods & the tables they touch

    
Routes follow the board's REST shape. JSON is camelCase at the API boundary; Postgres is snake_case (Pydantic alias mapping). sync returns immediately; async kicks off a background job and returns a job id. In this prototype the only async candidate is AI-assisted deliverable generation — and it runs computed/sync until the AI layer is added, at which point it becomes a job.
